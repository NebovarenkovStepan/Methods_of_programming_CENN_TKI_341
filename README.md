# Цифровая платформа "умного" больничного кластера

## Постановка задачи
Цифровая платформа «умного» больничного кластера — это единая система, которая включает в себя: электронные медицинские карты (ЭМК), поток данных пациентов, телемедицину (видеоконсультации), лабораторные исследования (заказы и результаты) и аптечную логистику (назначения, выдачи и учёт).

# Ценности
| **Ценности** | **Негативные события** | **Оценка** | **Комментарий** |
| :--- | :--- | :--- | :--- |
| Жизнь и здоровье пациента | Отправка неверных команд на медицинское оборудование (или изменение показателей), причиняющая вред здоровью или гибель пациента | Высокий | Система должна обеспечить целостность команды и данных. |
| Медицинская тайна | Массовая утечка денежных данных пациентов (ЭМК, результаты анализа, записи телемедицины) | Высокий | Грозит крупными штрафами, репутацией и исками |
| Своевременность оказания помощи | Из-за подмены статусов анализа или сбоя в аптечной логистике пациент не получил должного лечения | Высокий | Требует строгой проверки места и цепочки поставок внутри кластера |
| Доступность данных | Из-за отказа базы данных или портала врачи не могут получить доступ к ЭМК | Высокий | Можно временно перейти в ручной режим или локальные записи в отделениях. |

# Известные ограничения и условия
По условиям организаторов должна использоваться микросервисная архитектура и брокер сообщений для реализации асинхронной работы сервисов.
Из-за требований конфиденциальности мы не можем использовать сторонние API нейросетей

# Цели и Предположения безопасности
#### Цели безопасности
1. При любых обстоятельствах доступ к чтению электронных медицинских карт из базы данных имеют только авторизованные пользователи.
2. При любых обстоятельствах анализы данных могут передаваться только авторизованной лаборатории.
3. При любых обстоятельствах видеосеансы телемедицины могут проводиться только авторизованным врачом.
4. При любых обстоятельствах запись новых диагноз, назначение, заметках  в ЭМК  производится только авторизованным врачом.
5. При любых обстоятельствах на портале врача отображаются только аутентичные данные 
6. При любых обстоятельствах анализы пациентов могут передаваться только авторизованной лабораторией 
7. При любых обстоятельствах только аутентичные рецепты принимаются в аптеке 
8. При любых обстоятельствах только у аутентичных пациентов принимаются рецепты
9. При любых обстоятельствах аутентичный и авторизованный пациент имеет доступ к чтению ЭМК
10. При любых обстоятельствах аутентичный и авторизованный  врач получает доступ к ЭМК пациента

#### Предположения безопасности
1. Установлена ​​физическая безопасность серверов, оборудования, устройств пациентов и врачей.
2. Аутентичные сотрудники благонадёжны и обладают необходимой квалификацией (Персонал больницы работает добросовестно, не предоставляя свои учетные данные третьему врачу и не прошедший необходимую квалификацию.)

# Архитектура системы
## Контекст работы системы
* Платформа выступает связующим звеном между медицинским персоналом, пациентами и внешними сервисами. Взаимодействие пользователей с системой осуществляется через единый портал и выделенный сервис видеосвязи.
* Только технический специалист имеет физический доступ к серверу.

# Компоненты системы
| **Компонент** | **Назначение** |
|-----------|------------|
| 1. Портал | Осуществляет общее взаимодействие пользователей (пациентов, врачей, администраторов) с системой, контролирует права доступа и выступает единым шлюзом к данным |
| 2. База данных (БД) | Отвечает за запись, чтение и безопасное хранение электронных медицинских карт (ЭМК), телеметрии, рецептов и статусов анализов |
| 3. Видеосвязь | Предоставляет защищенный изолированный канал для проведения телемедицинских сеансов (видеоконсультаций) между врачом и пациентом |
| 4. Мед. оборудование | Осуществляет комплекс действий по сбору физиологических показателей пациента в режиме реального времени и первичной оцифровке этих данных |
| 5. Система мониторинга | Собирает и агрегирует потоковые метрики с медицинского оборудования перед их сохранением в базу данных |
| 6. Самодиагностика | Предоставляет данные оценки состояния медицинских приборов (например, контроль датчиков), блокируя передачу искаженных показателей |
| 9. LLM (Языковая модель) | Осуществляет анализ черновых записей врача и генерацию структурированных медицинских заключений на основе обезличенных данных |
| 10. Сервер | Обрабатывает запросы пользователей, выполняет бизнес-логику и координирует обмен данными с другими компонентами системы |
| 11. API | Обеспечивает единый интерфейс обмена данными между порталом и внешними либо внутренними подсистемами |
| 12. Веб-интерфейс | Основной пользовательский компонент, через который пациент, врач и администратор взаимодействуют с системой |
| 13. Логирование | Фиксирует действия пользователей, обращения к данным и события безопасности |
| 14. Аптека | Предоставляет статус проверки аутентичности рецептов и учет их выдачи авторизованным пациентам |
| 15. Локальная база данных | Хранит сведения о наличии лекарств, партиях, сроках годности, истории выдачи препаратов и статусах обработки рецептов|
| 16. Склад лекарственных препаратов| Представляет физическое хранилище медикаментов, в котором размещаются препараты, медицинские изделия и сопутствующие товары с учетом условий хранения |
| 17. Сканер штрихкодов/QR-кодов | Используется для считывания маркировки препаратов, идентификаторов рецептов и данных упаковок при отпуске и инвентаризации |
| 18. Аптечная информационная система | Основной программный компонент аптеки, обеспечивающий учет рецептов, обработку операций отпуска, контроль остатков и обмен данными с другими подсистемами больничного кластера |
| 19. Лаборатория | Отвечает за получение заявок на исследования от врачей и загрузку результатов анализов обратно в базу данных кластера |
| 20. Лабораторная информационная система | Основной программный компонент лаборатории, управляющий учетом исследований, маршрутизацией проб, сбором результатов и передачей данных в общую систему |
| 21. Лабораторные анализаторы | Представляют собой медицинское оборудование, выполняющее автоматическое исследование образцов и формирующее первичные результаты измерений|
| 22. Рабочие станции подключения оборудования | Обеспечивают связь анализаторов с лабораторной информационной системой и передачу результатов в цифровом виде |
| 23. Хранилище образцов | Предназначено для временного или длительного хранения биоматериалов до завершения исследований, повторной проверки или архивирования |

##### Сценарии нарушения целей безопасности

| Название сценария | Описание |
| --- | --- |
| НС-1 | Злоумышленник получил доступ к учетной записи пациента и просматривает ЭМК без авторизации, что нарушает ЦБ1, ЦБ8. |
| НС-2 | Злоумышленник получил доступ к учетной записи врача и изменяет медицинские данные пациента, что нарушает ЦБ4, ЦБ2, ЦБ9. |
| НС-3 | Передача данных ЭМК происходит по незащищенному каналу, что позволяет перехватить информацию, нарушая ЦБ1, ЦБ7. |
| НС-4 | Злоумышленник перехватывает API-запрос и подменяет данные пациента при передаче, нарушая ЦБ2, ЦБ5, ЦБ7. |
| НС-5 | Портал отображает искаженные данные пациента из-за атаки или ошибки, нарушая ЦБ5. |
| НС-6 | SQL-инъекция в API позволяет получить доступ к базе данных и изменить данные, нарушая ЦБ1, ЦБ2. |
| НС-7 | Неавторизованный пользователь получает доступ к видеосеансу телемедицины, нарушая ЦБ3. |
| НС-8 | Злоумышленник инициирует запись в ЭМК без прав врача, нарушая ЦБ4. |
| НС-9 | Лабораторная информационная система передает результаты в неавторизованную систему, нарушая ЦБ2, ЦБ6. |
| НС-10 | Подмена идентификатора биоматериала приводит к привязке анализа к неверному пациенту, нарушая ЦБ2, ЦБ6. |
| НС-11 | Удаление результатов анализов делает их недоступными врачу, нарушая ЦБ3. |
| НС-12 | Неавторизованный пользователь обращается к API для получения ЭМК или лабораторного исследования по чужому идентификатору и получает медицинские данные пациента, нарушая ЦБ1, ЦБ9. |
| НС-13 | Злоумышленник подменяет заявку на лабораторное исследование, что приводит к неверным анализам, нарушая ЦБ2, ЦБ6. |
| НС-14 | Злоумышленник от имени неавторизованной лаборатории отправляет или завершает результаты анализа пациента, и эти данные сохраняются в ЭМК или отображаются врачу, нарушая ЦБ5, ЦБ6. |
| НС-15 | Использование поддельного электронного рецепта (QR) в аптеке приводит к несанкционированной выдаче препарата, нарушая ЦБ7. |
| НС-16 | Повторное использование уже погашенного рецепта приводит к повторной выдаче лекарства, нарушая ЦБ7. |
| НС-17 | Изменение данных рецепта в системе приводит к выдаче неверного препарата, нарушая ЦБ7. |
| НС-18 | Неавторизованный пользователь создает запись ЭМК от имени врача через портал, добавляя пациенту ложные жалобы, назначения или заметки, нарушая ЦБ4, ЦБ5. |
| НС-19 | Неавторизованный пользователь изменяет существующую запись ЭМК пациента через API или прямую операцию сервиса, подменяя медицинские данные, нарушая ЦБ4, ЦБ5. |
| НС-20 | Злоумышленник выполняет XSS-атаку через веб-интерфейс и получает доступ к сессии врача, нарушая ЦБ1, ЦБ4. |
| НС-21 | Перехват cookie сессии позволяет злоумышленнику выполнять действия от имени пациента, нарушая ЦБ1, ЦБ8. |
| НС-22 | Злоумышленник отправляет поддельные API-запросы от имени клиента (Replay-атака), нарушая ЦБ2, ЦБ4. |
| НС-23 | Неавторизованный пользователь получает список лабораторных исследований пациента и узнает состав назначенных анализов без права доступа к ЭМК, нарушая ЦБ1, ЦБ9. |
| НС-24 | Неавторизованный пользователь создает лабораторное назначение от имени врача, после чего лаборатория принимает его как корректное назначение пациента, нарушая ЦБ4, ЦБ5, ЦБ6. |
| НС-25 | Видеосервис передает данные без шифрования, что позволяет перехватить видеосеанс, нарушая ЦБ1, ЦБ3. |
| НС-26 | Подмена сервера видеосвязи (Evil Twin) позволяет подключиться к фальшивому сеансу, нарушая ЦБ3, ЦБ4. |
| НС-27 | Аптека принимает рецепт без проверки его подлинности: злоумышленник передает поддельный QR/код рецепта, и система сохраняет событие сканирования как валидное, нарушая ЦБ7. |
| НС-28 | Злоумышленник получает рецепт по чужому идентификатору без авторизации и использует данные рецепта для получения лекарства, нарушая ЦБ7. |
| НС-29 | Неавторизованный пользователь создает или запускает видеосеанс телемедицины от имени врача, получая возможность провести консультацию с пациентом без подтверждения прав врача, нарушая ЦБ3. |
| НС-30 | Несогласованность данных между БД и ЛИС приводит к отображению устаревших результатов, нарушая ЦБ5. |
| НС-31 | Злоумышленник передает в аптеку рецепт с подмененным идентификатором пациента или препарата, и аптека принимает его как аутентичный, нарушая ЦБ7. |
| НС-32 | Неавторизованный пользователь создает лабораторное назначение от имени врача, после чего лаборатория получает и обрабатывает неаутентичное назначение анализа, нарушая ЦБ4, ЦБ5, ЦБ6. |
| НС-33 | Злоумышленник получает доступ к резервным копиям БД и извлекает медицинские данные, нарушая ЦБ1, ЦБ5. |
